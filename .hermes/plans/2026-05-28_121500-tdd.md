# Plano: TDD (RED → GREEN → REFACTOR) — Cratua Media Forge

**Data:** 2026-05-28
**Objetivo:** Introduzir TDD para evitar regressões e acelerar desenvolvimento.

---

## Situação Atual

- **Problema:** Features quebram outras features. Debug manual consome 80% do tempo.
- **Causa:** Zero testes automatizados. Toda validação depende de build + release + teste manual no Windows.
- **Meta:** Suite de testes que rode em < 10s localmente, cobrindo a lógica central.

---

## Estratégia: Pirâmide de Testes

```
        ┌──────┐
        │ E2E  │  1-2 testes de fluxo completo (opcional, pesado)
        ├──────┤
        │ INT  │  Testes de integração Rust (Tauri commands)
        ├──────┤
        │ UNIT │  Lógica pura: ffmpeg cmd builder, enums, params, JS puro
        └──────┘
```

---

```yaml
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: cargo test -p mediaforge-core
      - run: cd crates/mediaforge-tauri && npm test
```

## Benefício Esperado

- **Antes:** 1h de debug manual por feature quebrada
- **Depois:** `cargo test && npm test` em 5s revela regressão
- **Confiança:** Refatorar sem medo de quebrar a aba Simple

---

## Risco

- **Médio:** Extrair funções JS pode introduzir bugs se não for feito com cuidado
- **Mitigação:** Fazer em branch separada, rodar build completo antes de merge

# Code Review: Cratua Media Forge v0.3.23

Avaliação do estado atual do código, com problemas encontrados e melhorias
propostas. Foco em qualidade, manutenibilidade e UX — sem quebrar o que existe.

**Data:** 2026-05-30 | **123 testes Rust passando** | **4 testes JS passando**

---

## Problemas encontrados

### 🔴 Bloqueantes (devem ser corrigidos)

**1. Sem timeout no encoding thread — risco de hang eterno**
- `ffmpeg.rs`: loop de leitura stderr roda até ffmpeg morrer
- Se ffmpeg trava (stream de rede, disco cheio, pipe quebrado), o thread
  nunca retorna e o botão Cancel não vai resolver (processo zumbi)
- **Solução:** heartbeat de 30s — se `out_time_us` não mudar em 30s, mata o
  processo. Alternativa: `wait_timeout()` com `Duration`.

**2. Race condition em custom presets — sem lock**
- `load_custom_presets()` e `save_custom_presets()` não têm `Mutex`
- Improvável na prática (operações são sequenciais via Tauri), mas possível
  em encoding simultâneo
- **Solução:** `static PRESET_LOCK: Mutex<()>` como já existe para history

**3. `renderQueue` com `onclick` inline em innerHTML é XSS risk**
- `j.error` é inserido como atributo `title` com `replace(/"/g,'&quot;')`
- Isso é frágil: se o erro contiver `'` ou `&`, quebra o HTML
- **Solução:** usar `textContent` ou `dataset` em vez de atributo; ou
  sanitizar com `encodeURIComponent`

### 🟡 Importantes (reduzem bugs e melhoram DX)

**4. Código duplicado — preset save/delete (×2), browse (×2)**

```js
// Aparece 2x idêntico (simple + advanced browse)
on($('#btn-browse'),'click', async () => { /* 13 linhas */ });
on($('#a-btn-browse'),'click', async () => { /* 13 linhas idênticas */ });
```

- **Solução:** extrair `async function browseOutputDir()` e chamar de ambos
  os handlers. Mesmo para save/delete de presets.

**5. Lista de built-in IDs hardcoded em 3 lugares**

```js
const bi=['default','web-h264','web-h265','web-vp9','archive-h264',
  'audio-mp3','audio-aac','audio-opus','gif'];
```

No `btn-delete-preset`, `adv-btn-delete-preset`, e vai aparecer em
qualquer lugar que precise distinguir built-in de custom. Se adicionar
um preset, esquece de atualizar em algum lugar.

- **Solução:** `invoke('get_presets')` já retorna. Usar um campo `builtin: true`
  no preset, ou expor uma lista `invoke('get_builtin_preset_ids')`.
  Alternativa simples: exportar do Rust via um novo command.

**6. `updateCmdPreview` sem debounce — dispara em todo keypress**

```js
on(el,'input', () => { updateCmdPreview(); ... });
```

Digitar "C:/Users/me/video.mp4" no campo extra_args dispara 25 chamadas
`invoke('build_command_preview')`. Cada uma é um IPC call + formatação.

- **Solução:** debounce 300ms com `clearTimeout`/`setTimeout`. Padrão:
  ```js
  let cmdPreviewTimer;
  function debouncedCmdPreview() {
    clearTimeout(cmdPreviewTimer);
    cmdPreviewTimer = setTimeout(updateCmdPreview, 300);
  }
  ```

**7. `collectAdvParams` — sem validação, retorna valores crus do DOM**

Se o usuário digitar "abc" no campo width, `even()` retorna 1920
(silentemente). Se digitar "-5" no CRF, o range slider vai aceitar (min=0).
O backend tem Dragon Trap pra CRF/bitrate, mas outros campos passam direto.

- **Solução:** `collectAdvParams()` deve validar ranges:
  - width/height: 64-7680
  - channels: 1-8
  - threads: 0-32
  - CRF: 0-51
  Fora do range → clampar ao limite mais próximo e mostrar toast.

**8. Estado `S` mutável global — sem proteção contra race conditions**

Tauri invoc commands são async, eventos (`job:progress`) chegam assincronamente.
Se o usuário clicar "Add to Queue" enquanto um job está em progresso, ambos
tocam `S.jobs` e `S.files`. O event loop do browser é single-threaded então
não há race real, mas a complexidade de "quem mexe em quê" é alta.

- **Solução a longo prazo:** migrar para um reducer pattern ou pelo menos
  documentar quais funções tocam quais partes de `S`.

**9. Progress bar textual frágil**

```js
'[' + '█'.repeat(f) + '░'.repeat(w-f) + ']'
```

Block characters dependem da font e do encoding. No Windows com font
sistema, `█` pode renderizar como □ ou com largura errada.

- **Solução:** substituir por `<div>` com background CSS (já tem a classe
  `.progress-bar-fill` no CSS mas não está sendo usada na renderização).

### 🟢 Desejáveis (qualidade de vida e polimento)

**10. Settings modal incompleto**

O `Config` tem 7 campos mas o modal de settings só expõe 1 (`ffmpeg_path`).
Os outros (`language`, `theme`, `default_mode`, `output_dir`, `default_preset`,
`parallel_jobs`) existem no Rust mas são inacessíveis ao usuário.

- **Solução:** adicionar campos no modal de settings. Output dir é o mais
  útil (já existe no Browse mas não persiste de forma óbvia).

**11. Estilos CSS divididos entre `index.html` e `input.css`**

`<style>` no `<head>` (linhas 10-41 do index.html) contém layout,
tabs, form-grid, form-input, collapsible. Isso deveria estar no `input.css`
junto com os estilos de scrollbar e range.

- **Solução:** mover `<style>` do index.html para `input.css`. O HTML fica
  mais limpo e o Tailwind gerencia a ordem de carregamento.

**12. Sem dark/light mode**

Tema é hardcoded dark (`class="dark"` no `<html>`, cores `#0b0b12` etc).
O enum `Theme` existe no Rust (`Dark`, `Light`), mas não há toggle.

- **Solução:** variáveis CSS (`--bg-primary`, `--text-primary`, etc.),
  toggle no header ou settings. Tailwind v4 suporta `light-dark()` e
  `prefers-color-scheme`.

**13. `check.sh` não testa `mediaforge-tauri`**

Roda `cargo test -p mediaforge-core` e `node --test` mas não roda os
testes do Tauri crate. Os testes existem (crate-type inclui `lib`).

- **Solução:** adicionar `cargo test -p mediaforge-tauri` ao `check.sh`.
  Precisa garantir que os testes não tentam abrir janela.

**14. `mediaforge-gui` (egui) legado ainda no workspace**

A crate egui não é mais usada desde a migração para Tauri. Fica no
workspace, compila com `cargo check --workspace`, adiciona ~30s ao build.

- **Solução:** remover do workspace `members` e arquivar (ou deletar).
  Se precisar referência futura, está no git history.

**15. Sem `.editorconfig` ou formatação automática**

Rust tem `rustfmt.toml`? JS não tem prettier config. HTML tem indentação
inconsistente (tabs vs espaços nos templates JS dentro de innerHTML).

- **Solução:** adicionar `.editorconfig` básico. Não precisa de prettier
  (esbuild já minifica), mas consistência ajuda.

**16. Sem CHANGELOG.md**

20 releases (0.3.4 → 0.3.23) sem changelog. Usuário baixa versão nova e
não sabe o que mudou.

- **Solução:** `CHANGELOG.md` no repositório, atualizado manualmente a
  cada release. Ou gerado do git log com `git-cliff`.

**17. Sem atalhos de teclado documentados**

O app tem atalhos (Ctrl+O, Ctrl+Enter, Escape) mas não são listados em
lugar nenhum. O usuário precisa adivinhar ou ler o código.

- **Solução:** adicionar seção "Keyboard Shortcuts" no modal About ou
  tooltip no header.

**18. Output path não mostra conflito visualmente**

Se `output == input` (mesmo arquivo), o app renomeia automaticamente para
`_converted`. Mas o usuário não vê isso acontecendo — só uma mensagem na
status bar que some em 5s.

- **Solução:** mostrar um ícone ⚠ ou tooltip no item da queue quando o
  nome foi alterado automaticamente.

## O que está bom (não mexer)

- **Core (ffmpeg.rs):** 825 linhas mas bem estruturado, 123 testes, cobre
  todos os codecs, containers, edge cases. O stderr loop é robusto.
- **Preset system:** Serialização limpa, built-in vs custom com guarda de ID,
  merge no `get_presets()`. Sólido.
- **Cancellation:** `Arc<AtomicBool>` + catch_unwind + limpeza do flag.
  Implementação correta e testada.
- **Diagnostic bar:** O `diag()` no frontend é a melhor decisão de
  debugging do projeto. Não remover.
- **Build pipeline:** `release.sh`, `download-ffmpeg.sh`, `check.sh` bem
  organizados. Cross-compile Windows do Linux funcional.
- **Pitfall tracking:** 55 pitfalls documentados no skill media-forge.
  Cultura de aprendizado com erros é excelente.

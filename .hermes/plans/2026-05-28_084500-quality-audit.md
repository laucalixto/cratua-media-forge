# Auditoria de Qualidade — Cratua Media Forge

**Data:** 2026-05-28
**Escopo:** Fluxo completo UI → Rust → ffmpeg → output
**Método:** Revisão de código (read-only) de `main.js`, `commands.rs`, `ffmpeg.rs`

---

## Bugs Críticos (impedem funcionamento correto)

### C1. `progress_pct` sempre 0 — barra de progresso nunca move

**Arquivo:** `crates/mediaforge-core/src/ffmpeg.rs` linhas 342-355

O callback `on_progress` no loop de leitura do stderr extrai `out_time_us`, `speed`, `fps`, `frame`, mas **nunca preenche `progress_pct`**. O campo fica com o valor default `0.0`.

```rust
// Linha 343: ProgressInfo::default() — progress_pct = 0.0
let mut info = ProgressInfo::default();
match key.as_str() {
    "out_time_us" => info.out_time_us = value.parse().unwrap_or(0),
    "speed" => ...,
    "fps" => ...,
    "frame" => ...,
    // FALTA: progress_pct
}
```

**Impacto:** Frontend mostra barra de progresso sempre em 0%. Usuário vê "running" sem feedback visual.

**Correção:** Usar `out_time_us` para estimar progresso comparando com a duração total (obtida via ffprobe), ou usar `speed` × tempo decorrido para estimativa.

---

### C2. Separador de caminho inconsistente no Windows

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js` linhas 305-312

`dialogOpen` retorna caminhos com `\` no Windows. O template string `${S.outputDir}/${stem}.${ext}` usa `/`. Resultado: caminhos híbridos `C:\Users\pasta/file.mp4`.

A comparação `output === f` na linha 311 **nunca será true** porque:
- `f` = `W:\Nova pasta\01.mp4` (backslash do dialog)
- `output` = `W:\Nova pasta/01.mp4` (slash do template)

**Impacto:** Detecção de colisão output=input inoperante no Windows. Caminhos com `/` e `\` misturados podem confundir o ffmpeg no Windows.

**Correção:** Normalizar todos os caminhos para o separador nativo ou usar `Path` API do Rust para validação (já existe no Rust, mas o JS precisa normalizar antes de enviar).

---

### C3. `navigator.platform` depreciado

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js` linha 479

```js
navigator.platform.includes('Win')
```

`navigator.platform` está depreciado e pode retornar string vazia ou `"Linux x86_64"` em WebViews modernos.

**Impacto:** No Windows, fallback do outputDir pode ir para `/home/user` (Linux) em vez de `C:\Users\Public`.

**Correção:** Usar `navigator.userAgentData?.platform` ou passar o OS do Rust via comando Tauri.

## Bugs Altos (comportamento incorreto)

### H1. Filtros e metadata compartilhados entre jobs (shallow copy)

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js` linha 319

```js
params: { ...baseParams }
```

O spread `...` faz shallow copy. Arrays como `video_filters`, `audio_filters`, `movflags`, `extra_args` e o objeto `metadata` são **compartilhados por referência** entre todos os jobs na fila.

**Impacto:** Se o usuário modificar filtros após adicionar à fila, todos os jobs pendentes são afetados. Pior: se o Rust modificar (não modifica), corromperia todos.

**Correção:** Deep clone com `structuredClone(baseParams)`.

---

### H2. Sem verificação de arquivo de saída existente

**Arquivos:** `main.js` linha 316, `ffmpeg.rs` build_command

Nenhum código verifica se o arquivo de saída **já existe no disco** (de uma conversão anterior ou outro arquivo fonte). O ffmpeg `-y` sobrescreve silenciosamente.

**Impacto:** Usuário perde arquivos sem aviso.

**Correção:** Adicionar comando Rust `check_output_exists` chamado antes de iniciar encoding, ou verificar no `start_encoding` e retornar lista de conflitos para o frontend exibir diálogo de confirmação.

---

### H3. Vazamento do cancel flag em caso de panic

**Arquivo:** `commands.rs` linhas 303-309

O cancel flag é limpo no final do `std::thread::spawn`. Se a thread **panicar** (ex: `unwrap()` em código futuro), o flag nunca é limpo. O `OnceLock` mantém o valor travado.

**Impacto:** Encoding futuros são cancelados imediatamente (flag fica `true`). Requer reiniciar o app.

**Correção:** Envolver corpo da thread em `std::panic::catch_unwind` com cleanup no `finally`, OU usar `Drop` guard, OU usar `tokio::spawn` com Tauri async runtime.

---

## Bugs Médios (UX e edge cases)

### M1. Preset selector não aplica parâmetros

**Arquivo:** `main.js` linha 493

O `<select id="preset-select">` é populado com presets, mas não há `onchange` handler que atualize `S.params` com os parâmetros do preset selecionado.

**Impacto:** Selecionar "Web Video (VP9)" continua usando parâmetros padrão H264.

**Correção:** Adicionar handler `onchange` que busca o preset selecionado e aplica `S.params = preset.params`.

---

### M2. Evento de progresso casa com job errado

**Arquivo:** `main.js` linhas 403-407

```js
const job = S.jobs.find(j => j.status === 'running' || j.status === 'pending');
```

Usa `Array.find()` que retorna o **primeiro** job matching. Se houver múltiplos jobs em paralelo (feature futura), eventos do job B atualizariam o job A.

**Impacto:** Baixo agora (1 job por vez), mas quebraria com paralelismo.

**Correção:** Usar `job_id` do payload: `S.jobs.find(j => j.id === data.job_id)`. Precisa incluir `id` no JobData e propagá-lo nos eventos.

---

### M3. Condição de corrida no histórico

**Arquivo:** `commands.rs` linhas 293-300

```rust
let mut history = load_history();  // LÊ
history.push(entry);               // MODIFICA
save_history(&history);            // ESCREVE
```

Sem lock, se dois jobs terminarem simultaneamente (threads diferentes), uma entrada pode sobrescrever a outra (lost update).

**Impacto:** Perda silenciosa de entradas de histórico.

**Correção:** Adicionar `Mutex<()>` global para serializar acesso ao arquivo de histórico.

---

### M4. `selected[0]` desnecessário no dialog directory

**Arquivo:** `main.js` linha 204

```js
S.outputDir = typeof selected === 'string' ? selected : selected[0];
```

Com `multiple: false` e `directory: true`, Tauri v2 sempre retorna `string`, nunca array. O fallback `selected[0]` nunca é usado mas confunde.

**Impacto:** Código morto. Baixo.

**Correção:** Simplificar para `S.outputDir = selected`.

---

## Bugs Baixos (cosméticos e resiliência)

### L1. Config save ignora falhas silenciosamente

**Arquivo:** `main.js` linha 206

```js
invoke('save_config', { config: S.config }).catch(() => {});
```

Falha ao salvar config é completamente ignorada.

---

### L2. Campo de path sem validação

**Arquivo:** `main.js` linha 214

Qualquer texto colado no `#path-input` é aceito como caminho de arquivo, sem verificar se é um caminho válido ou se o arquivo existe.

---

### L3. `EncodeParams` default `threads: 0` sem tratamento especial

Threads=0 significa "auto" para ffmpeg, mas não é documentado na UI. O campo no advanced mode mostra 0 como valor inicial.

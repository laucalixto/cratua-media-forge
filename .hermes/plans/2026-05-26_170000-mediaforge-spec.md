# MediaForge — Spec de Aplicativo

**Data:** 2026-05-26
**Autor:** @laucalixto
**Stack:** Rust + egui/eframe
**Alvo:** Linux (AppImage/deb) + Windows (.exe portátil)

---

## 1. Objetivo

Criar um aplicativo desktop portátil, cross-platform, com interface gráfica, que encapsula o ffmpeg para conversão e otimização de mídia (vídeo/áudio) para web.

O ponto de partida é o script `context/RenderConverterFFMPEG_v.0.0.4_deinterlace.bat`, que hoje faz conversão batch de forma textual e manual, com parâmetros fixos (libx264, yadif, scale, crf, audio bitrate).

---

## 2. Análise do script original

### 2.1 O que o .bat faz

| Etapa | Descrição |
|-------|-----------|
| Config | Prompt interativo: largura, altura, qualidade CRF (0-51, default 19), qualidade áudio kbps (96/128/192/320, default 128) |
| Confirmação | Mostra os valores e oferece reconfigurar, iniciar ou sair |
| Render | Cria pasta `low/`, converte **todos** os arquivos do diretório corrente com ffmpeg |
| Pós-render | Mensagem de conclusão, opção de reconfigurar ou sair |

### 2.2 Comando ffmpeg usado

```
ffmpeg -i "%%a" \
  -c:v libx264 \
  -crf %QUALIDADE% \
  -vf yadif,scale=%LARGURA%:%ALTURA% \
  -pix_fmt yuv420p \
  -movflags faststart \
  -threads 2 \
  -b:a %QUALIDADEAUDIO%k \
  "low\%%~na.mp4"
```

### 2.3 Limitações do script

- Sem seleção de codec (fixo libx264)
- Sem seleção de container (fixo mp4)
- Sem preset de encoding
- Sem controle de framerate
- Sem filtros avançados (crop, rotate, denoise, etc.)
- Sem codec de áudio customizável (usa padrão do mp4)
- Batch cego: processa todos os arquivos do diretório, sem seleção individual
- Sem preview, sem fila, sem progresso individual
- Interface textual DOS, não portátil (Windows only)
- Sem i18n (só português)

> Um utilitário desktop para converter e otimizar mídia para web com ffmpeg. Interface gráfica bilingue, dois modos de uso (simples e avançado), portátil — sem instalação, sem dependências manuais.

## 5. Arquitetura Técnica

### 5.1 Stack

| Camada | Tecnologia | Justificativa |
|--------|-----------|---------------|
| Linguagem | Rust | Performance, segurança de memória, cross-compilation madura, binário único |
| GUI | **egui + eframe** | Immediate mode, nativa (wgpu/glow), sem webview, ~2-5 MB binário. Alternativa: Iced (Elm-like, mais verbosa, mas nativa) |
| i18n | `fluent-rs` (Project Fluent) | Padrão Mozilla, plural rules, suporte a gênero, maduro |
| Config | `serde` + `TOML` | Legível, tipado, padrão Rust |
| Processos | `tokio` + `std::process::Command` | Async para não travar UI durante encoding |
| Bundling ffmpeg | `rust-embed` (embed binário) ou detecção em PATH | Portabilidade: incluir ffmpeg estático compilado |
| Testes | `cargo test` + mock de ffmpeg | Testes unitários e de integração |
| CI/CD | GitHub Actions | Build matrix: Linux x86_64 + Windows x86_64. Release via GH Releases |

### 5.3 Modelo de dados principal

```rust
// job.rs
struct Job {
    id: Uuid,
    input_path: PathBuf,
    output_path: PathBuf,
    params: EncodeParams,
    status: JobStatus,   // Pending, Running(f64), Completed, Failed(String), Cancelled
    ffmpeg_command: String,
    duration: Option<Duration>,
    created_at: Instant,
}

struct EncodeParams {
    // Video
    video_codec: VideoCodec,       // H264, H265, VP9, AV1, Copy
    width: u32,
    height: u32,
    scale_algorithm: ScaleAlgo,    // Bilinear, Bicubic, Lanczos
    fps: FpsMode,                  // SameAsSource, Fixed(u32)
    crf: Option<u8>,               // 0-51
    video_bitrate: Option<u32>,    // kbps
    max_bitrate: Option<u32>,
    bufsize: Option<u32>,
    preset: Preset,                // Ultrafast..Veryslow
    profile: Profile,              // Baseline, Main, High
    pixel_format: PixelFormat,     // Yuv420p, etc.
    deinterlace: Option<DeinterlaceMethod>,
    video_filters: Vec<VideoFilter>,

    // Audio
    audio_codec: AudioCodec,       // Aac, Mp3, Opus, Copy
    audio_bitrate: u32,            // kbps
    audio_channels: u8,
    sample_rate: u32,
    audio_filters: Vec<AudioFilter>,

    // Output
    container: Container,          // Mp4, Mkv, Webm, Mov, Gif
    movflags: Vec<MovFlag>,        // FastStart, FragKeyframe
    threads: u8,                   // 0 = auto

    // Metadata
    metadata: HashMap<String, String>,
    trim_start: Option<String>,    // -ss
    trim_end: Option<String>,      // -to
}
```

### 5.4 Fluxo de encoding

```
User seleciona arquivos → Configura parâmetros → Adiciona à fila → Inicia batch
                                                                    ↓
Para cada job:                                                    tokio::spawn
  1. Constrói comando ffmpeg (EncodeParams → Vec<String>)
  2. Spawn subprocess, captura stderr (progresso)
  3. Parser extrai frame atual, fps, bitrate, tempo
  4. Envia via channel para GUI atualizar barra de progresso
  5. On exit: status Completed ou Failed(stderr tail)
  6. Próximo job (serial ou paralelo configurável)
```

## 7. Riscos e Tradeoffs

| Risco | Mitigação |
|-------|-----------|
| egui é immediate mode — performance com muitos widgets? | egui é eficiente, mas modo avançado terá scroll containers e collapsible sections |
| Parsing de progresso do ffmpeg é frágil (varia por codec) | Regex tolerante, fallback para "sem progresso" (apenas spinner) |
| ffmpeg ausente no sistema do usuário | Oferecer download/bundle de binário estático (ffmpeg ~80 MB) |
| Cross-compilation Rust + GUI pode ser complicada | Usar cross + CI com runners nativos Windows/Linux |
| Complexidade do modo avançado assusta usuário simples | O modo simples é o default; avançado é opt-in com aviso |

## 8. Perguntas em Aberto

1. **Preview visual de frame?** Gerar um thumbnail do primeiro frame com filtros aplicados? Custo: +1 spawn de ffmpeg por job antes de iniciar. Valor: alto para ajuste visual. → Deixar para fase futura.

2. **Processamento paralelo?** Vários arquivos simultaneamente vs sequencial. Paralelo satura CPU mas é mais rápido. → Configurável (default: sequencial, opção "paralelo: N threads").

3. **Licença?** MIT ou GPL? ffmpeg é LGPL/GPL dependendo dos codecs compilados. → Sugiro MIT para o app, com nota de que ffmpeg bundled segue sua própria licença.

4. **Nome final?** MediaForge é sugestão. Validar disponibilidade (crates.io, GitHub, domínio).

## 9. Referência: Paridade com o .bat original

O modo simples do MediaForge deve reproduzir, como preset "Web H.264 (Legacy)", exatamente o comando do script original:

```
ffmpeg -i <input> \
  -c:v libx264 \
  -crf <quality> \
  -vf yadif,scale=<w>:<h> \
  -pix_fmt yuv420p \
  -movflags faststart \
  -threads 2 \
  -b:a <audio_quality>k \
  <output>.mp4
```

Isso garante backward compatibility com o workflow do usuário e serve como validação funcional (o app produz outputs idênticos ao script).

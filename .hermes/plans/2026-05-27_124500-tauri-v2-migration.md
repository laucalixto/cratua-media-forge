# Plano — Migração Cratua Media Forge para Tauri v2 + Tailwind

**Data:** 2026-05-27
**Objetivo:** Substituir egui por interface web moderna (Tauri v2 + Tailwind CSS), com arquitetura que permita futura migração para WebAssembly/web online.

---

## 1. Estratégia multi-target (desktop → web)

A arquitetura é desenhada para que o mesmo frontend (HTML + Tailwind + JS) funcione em 3 destinos sem reescrita:

```
┌─────────────────────────────────────────────┐
│              UI Layer (HTML/Tailwind/JS)    │  ← mesmo código em todos
├──────────────┬──────────────┬───────────────┤
│  Tauri (IPC) │  WASM (FFI)  │  HTTP (REST)  │  ← camada de comunicação
├──────────────┼──────────────┼───────────────┤
│  Rust native │  Rust WASM   │  Rust server  │  ← backend (mesmo core)
├──────────────┼──────────────┼───────────────┤
│  ffmpeg nativo │  ❌ sem ffmpeg │ ffmpeg server │
└──────────────┴──────────────┴───────────────┘
```

- **Tauri desktop:** Rust nativo + ffmpeg bundled, IPC via `#[tauri::command]`
- **WebAssembly (futuro):** `mediaforge-core` compilado para WASM, frontend via fetch/FFI, sem ffmpeg local (usaria browser APIs ou server-side)
- **Web online (futuro):** mesmo frontend hospedado, backend Rust via API REST, ffmpeg no servidor

## 2. Stack tecnológica

### Backend (Rust)
| Camada | Tecnologia |
|--------|-----------|
| Desktop | Tauri v2 |
| IPC | `#[tauri::command]` + `#[tauri::event]` (eventos de progresso) |
| Core | `mediaforge-core` (zero alterações) |
| WASM target | `wasm-pack` + `wasm-bindgen` (futuro) |
| Config | `serde` + `TOML` (mesmo `Config`) |

### Frontend (Web)
| Camada | Tecnologia |
|--------|-----------|
| Framework CSS | **Tailwind CSS v4** (utility-first, rápido, ecossistema enorme) |
| JavaScript | Vanilla JS (sem framework — máximo de portabilidade) |
| Bundler | `esbuild` (leve, rápido, zero-config para vanilla) |
| Ícones | Lucide Icons (SVG inline, ~1KB cada) |
| Fontes | Inter (sans-serif) + JetBrains Mono (monospace para comandos) |

## 4. IPC Design (Rust ↔ Frontend)

### Commands (Frontend → Rust)
```rust
#[tauri::command] fn get_presets() -> Vec<Preset>
#[tauri::command] fn detect_ffmpeg(path: Option<String>) -> Option<String>
#[tauri::command] fn get_config() -> Config
#[tauri::command] fn save_config(config: Config) -> Result<(), String>
#[tauri::command] fn build_command_preview(params: EncodeParams) -> String
#[tauri::command] fn get_i18n(lang: String, key: String) -> String
#[tauri::command] async fn start_encoding(jobs: Vec<JobData>, ffmpeg_path: Option<String>, window: Window)
#[tauri::command] fn cancel_encoding()
```

### Events (Rust → Frontend)
```rust
#[derive(Clone, Serialize)]
struct ProgressEvent { job_id: String, progress_pct: f64, speed: f64, fps: f64, out_time_us: u64 }
#[derive(Clone, Serialize)]
struct JobDoneEvent { job_id: String, status: String, error: Option<String> }
```

### Fluxo de encoding
```
1. Frontend → start_encoding(jobs, ffmpeg_path)
2. Rust spawna thread para cada job (sequencial)
3. A cada progresso: window.emit("job:progress", ProgressEvent)
4. Ao final: window.emit("job:done", JobDoneEvent)
5. Frontend escuta eventos com window.listen() e atualiza UI
6. Cancel: frontend chama cancel_encoding(), Rust seta AtomicBool + Child::kill()
```

## 6. Cancel real (design Tauri)

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::{Arc, Mutex};

struct CancelState {
    cancelled: Arc<AtomicBool>,
    current_child: Arc<Mutex<Option<std::process::Child>>>,
}

#[tauri::command]
fn cancel_encoding(state: tauri::State<'_, CancelState>) {
    state.cancelled.store(true, Ordering::SeqCst);
    if let Some(child) = state.current_child.lock().unwrap().take() {
        let _ = child.kill();
        let _ = child.wait(); // recolher processo zumbi
    }
}
```

## 9. Caminho para WebAssembly (futuro)

Quando o desktop estiver estável, o mesmo frontend pode ir pra web:

### Opção A: WASM (tudo no browser)
```
1. Compilar mediaforge-core com wasm-pack
2. Expor funções via wasm-bindgen
3. Frontend chama WASM em vez de Tauri IPC
4. Limitação: sem ffmpeg nativo → usar browser APIs para conversão simples
```

### Opção B: Servidor Rust + Frontend Web
```
1. Criar crate mediaforge-server (Actix-Web / Axum)
2. Frontend chama REST API em vez de Tauri IPC
3. ffmpeg roda no servidor
4. Upload/download de arquivos
```

**O frontend não muda** — só a camada de bridge (`tauri-bridge.js` → `wasm-bridge.js` ou `http-bridge.js`).

## 10. Riscos

| Risco | Mitigação |
|-------|-----------|
| Tailwind v4 é recente (breaking changes vs v3) | Pin da versão no package.json |
| WebView Linux precisa de `libwebkit2gtk` | Documentar, verificar no build |
| Bundle maior (~5MB + WebView) vs egui (5MB total) | Aceitável pelo ganho visual |
| Complexidade IPC vs código direto | Commands simples, estado mínimo |
| Perda do egui se Tauri der problema | Manter `mediaforge-gui` compilável no workspace |

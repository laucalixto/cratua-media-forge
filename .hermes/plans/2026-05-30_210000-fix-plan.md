### 1. ⏱️ Timeout no encoding thread

**Arquivo:** `crates/mediaforge-core/src/ffmpeg.rs`
**Linhas:** 443-485 (stderr loop)

**Problema:** Se ffmpeg trava (disco cheio, pipe quebrado, stream de rede),
o loop `for line in reader.lines()` nunca termina. O Cancel mata o processo
se houver output, mas sem output o `lines()` bloqueia pra sempre.

**Solução:** Adicionar `child.wait_timeout(Duration::from_secs(30))` em
uma thread separada que monitora o processo. Se o processo não terminar
em 30s, mata.

**Alternativa mais simples:** Heartbeat no loop — se `out_time_us` não
mudar por 30s, assume hang e mata.

**Implementação (heartbeat):**

```rust
// No início do loop, antes de for line in reader.lines():
let heartbeat_start = std::time::Instant::now();
let mut last_out_time_us: u64 = 0;
let mut heartbeat_strikes: u8 = 0;
const HEARTBEAT_MAX_STRIKES: u8 = 3;  // 3 × ~10s = 30s

for line in reader.lines() {
    // ... existing cancel check ...

    if let Some((key, value)) = parse_progress_line(&line) {
        if key == "out_time_us" {
            let current = value.parse::<u64>().unwrap_or(0);
            if current == last_out_time_us && current > 0 {
                // ffmpeg pode estar travado (não avança)
                // ou pode estar em uma pausa legítima (codec lento)
                // Só mata após N strikes consecutivos
            } else {
                last_out_time_us = current;
                // reseta strikes quando o progresso avança
            }
        }
    }
}
```

Na verdade, essa abordagem é frágil — `reader.lines()` bloqueia na leitura,
não no processamento. Se ffmpeg não produz output, o loop bloqueia ANTES
de chegar no nosso código.

**Abordagem correta:** `wait_timeout` com thread separada.

```rust
// Depois de spawnar o child, antes de começar a ler stderr:
let mut child = cmd.spawn()...;

// Spawn a watchdog thread
let child_id = child.id();
let watchdog = std::thread::spawn(move || {
    // Wait for child to exit with 30s timeout per check
    // This runs in parallel with stderr reading
    // If stderr reading finishes first, this thread is abandoned (no JoinHandle held)
});
```

Na verdade, a abordagem mais limpa é um `select!`-like pattern usando
`std::sync::mpsc`:

```rust
let (tx, rx) = std::sync::mpsc::channel();
let stderr_thread = std::thread::spawn(move || {
    let reader = std::io::BufReader::new(stderr);
    for line in reader.lines() {
        // ... existing logic ...
        if tx.send(()).is_err() { break; } // heartbeat ping
    }
});

// Main loop waits for either stderr lines or timeout
loop {
    match rx.recv_timeout(Duration::from_secs(30)) {
        Ok(()) => { /* line was processed, keep waiting */ }
        Err(std::sync::mpsc::RecvTimeoutError::Timeout) => {
            // 30s without a line → ffmpeg hung
            let _ = child.kill();
            return Err(MediaForgeError::FfmpegProcess("ffmpeg timed out (no output for 30s)".into()));
        }
        Err(std::sync::mpsc::RecvTimeoutError::Disconnected) => {
            break; // stderr thread finished normally
        }
    }
}
```

Mas isso complica o BufReader + callback on_progress. A abordagem mais
simples que não quebra nada:

**Abordagem final (simples, testável):** `child.try_wait()` a cada iteração
do loop, com um contador de iterações sem progresso.

```rust
let mut last_progress_line: u64 = 0;  // contador de linhas desde último progresso
const MAX_IDLE_LINES: u64 = 500_000;  // ~30s assumindo ~60μs/line

for line in reader.lines() {
    last_progress_line += 1;

    // Heartbeat: check child status every 100k lines
    if last_progress_line > 0 && last_progress_line % 100_000 == 0 {
        match child.try_wait() {
            Ok(Some(status)) => { /* process exited, break loop */ break; }
            Ok(None) => { /* still running, continue */ }
            Err(e) => { return Err(...); }
        }
    }

    // ... existing logic, reset counter on progress ...
}
```

Isso é frágil também. Vou usar a abordagem mais robusta: **channel-based**.

**Plano de implementação:**

1. Criar uma thread para ler stderr que envia cada linha por channel
2. Thread principal faz `rx.recv_timeout(30s)`
3. Se timeout → `child.kill()` + erro
4. Se channel fechado → stderr terminou, `child.wait()` pelo exit code

Isso encapsula bem. O `on_progress` callback é chamado da thread principal.

**RED (teste):** Adicionar teste que verifica timeout com stderr que nunca
produz output (simular com pipe vazio).

### 3. 🛡️ Sanitização do renderQueue

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js`
**Linhas:** ~211 (renderQueue)

**Problema:** `innerHTML` com `j.error` no atributo `title` + `onclick` inline.
Se o erro contiver aspas simples ou `&`, quebra o HTML. O replace de `"`
por `&quot;` não é suficiente.

**Solução:** Não usar `innerHTML` para dados não confiáveis. Construir o
elemento com `createElement` + `textContent` para a mensagem de erro.

**Implementação:**

```js
// Caso 'failed' no renderQueue — substituir innerHTML por:
case 'failed': {
    const div = document.createElement('div');
    div.className = 'flex flex-col text-xs py-1 px-2';

    const row = document.createElement('div');
    row.className = 'flex items-center justify-between';
    row.innerHTML = `<span class="text-[#ef4444] truncate">✗ ${inn}</span>
      <span class="text-[#ef4444] shrink-0 ml-2">failed</span>`;

    const errDiv = document.createElement('div');
    errDiv.className = 'text-[#606070] mt-1 text-[10px] cursor-pointer hover:text-[#c0c0d0]';
    errDiv.textContent = (j.error || 'unknown').substring(0, 500);
    errDiv.title = 'Click to copy';
    errDiv.addEventListener('click', () => {
        navigator.clipboard.writeText(j.error || '');
    });

    div.appendChild(row);
    div.appendChild(errDiv);
    return div.outerHTML; // ok here — div structure is controlled
}
```

**Melhor ainda:** Extrair para função `renderJobRow(j)` que retorna DOM
element, e `renderQueue` usa `appendChild` em vez de `innerHTML`. Isso
elimina todo o risco de XSS de uma vez.

**Impacto:** Mudança maior (reescrever `renderQueue` de template string
para DOM manipulation), mas mais segura. Manter compatibilidade com
a atualização direta do DOM no handler de `job:progress`.

**Abordagem pragmática (só o fix de sanitização):**
Manter `innerHTML` mas sanitizar `j.error` com `encodeURIComponent` ANTES
de interpolar:

```js
const safeError = encodeURIComponent(j.error || 'unknown').substring(0, 500);
// No template: ... title="${safeError}" ...
```

E remover o `onclick` inline, substituir por event delegation no
`#queue-list`:

```js
// Depois de renderQueue():
$('#queue-list').addEventListener('click', (e) => {
    const el = e.target.closest('.error-msg');
    if (el && el.dataset.error) {
        navigator.clipboard.writeText(decodeURIComponent(el.dataset.error));
    }
});

// No template: <div class="error-msg" data-error="${safeError}">...</div>
```

**RED (teste):** Adicionar teste JS em `test.js`:
```js
describe('error sanitization', () => {
    it('handles quotes in error message', () => {
        const err = `Error: can't "parse" this & that`;
        const safe = encodeURIComponent(err);
        assert(!safe.includes("'"));
        assert(!safe.includes('"'));

```

### 7. ✅ Validação de ranges no collectAdvParams

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js`
**Linhas:** 116-141

**Problema:** Campos numéricos sem validação. "abc" no width → `even()` retorna
1920 silenciosamente. "-5" no CRF → aceito (min=0 no HTML mas pode vir de
programação).

**Solução:** Função helper `clamp(val, min, max, fallback)` e aplicar em
todos os campos numéricos.

```js
function clamp(val, min, max, fallback) {
    const n = parseInt(val);
    if (isNaN(n)) return fallback;
    return Math.max(min, Math.min(max, n));
}

function collectAdvParams() {
    // ... existing code, com validação:
    const width = even(clamp($('#a-width').value, 64, 7680, 1920));
    const height = even(clamp($('#a-height').value, 64, 4320, 1080));
    const crf = clamp($('#a-crf').value, 0, 51, 23);
    const ab = clamp($('#a-abitrate').value, 32, 320, 128);
    const ch = clamp($('#a-channels').value, 1, 8, 2);
    const sr = clamp($('#a-samplerate').value, 8000, 96000, 48000);
    const thr = clamp($('#a-threads').value, 0, 32, 0);
    // ...
}
```

**Toast de aviso:** Se o valor foi clampado (diferente do digitado), mostrar
diag amarelo:

```js
function validateAndWarn(id, val, clamped) {
    if (val !== clamped) {
        diag(`⚠ ${id} adjusted: ${val} → ${clamped}`);
    }
    return clamped;
}
```

**RED (teste):** Adicionar à `test.js`:
```js
describe('clamp()', () => {
    it('clamps within range', () => assert.equal(clamp(50, 0, 100, 0), 50));
    it('clamps below min', () => assert.equal(clamp(-5, 0, 100, 0), 0));
    it('clamps above max', () => assert.equal(clamp(999, 0, 100, 0), 100));

```

### 18. ⚠️ Indicador visual de rename automático na queue

**Arquivo:** `crates/mediaforge-tauri/ui/src/main.js`
**Linhas:** 210 (btn-add-queue) e 211 (renderQueue)

**Problema:** Quando `output == input`, o app renomeia para `_converted`
mas o usuário só vê uma mensagem na status bar que some em 5s.

**Solução:** Adicionar campo `renamed: true` no job e mostrar ícone ⚠ no
item da queue.

**No btn-add-queue (linha 210):**

```js
S.files.forEach(f => {
    const nm = f.split('/').pop();
    const st = nm.includes('.') ? nm.substring(0, nm.lastIndexOf('.')) : nm;
    let out = `${S.outputDir}/${st}.${ext}`;
    let renamed = false;
    if (out === f) {
        out = `${S.outputDir}/${st}_converted.${ext}`;
        renamed = true;
    }
    S.jobs.push({
        input: f, output: out, params: structuredClone(bp),
        status: 'pending', progress: 0, renamed
    });
});
```

**No renderQueue — caso 'pending':**

```js
case 'pending':
    const rnIcon = j.renamed ? ' ⚠' : '';

      <span class="text-[#606070] shrink-0 ml-2">pending</span>
    </div>`;
```
